# Generators & Commands

This repository includes interactive generators to standardize module code structure.

## Create a module

- `php artisan m:module Admin`

This creates `Modules/Admin/*` and prefixed base classes for the module.

Default test scaffold after `m:module`:

- removes `Tests/Feature/ExampleTest.php` and `Tests/Unit/ExampleTest.php`
- generates `Tests/Feature/ModuleRouteFilesTest.php`
- generates `Tests/Feature/RouteRegistrationSmokeTest.php`

Useful options:

- `--api` create API-only module (default behavior in this repo unless `--with-web`)
- `--with-web` keep `Routes/web.php`

## Create a controller (interactive wizard)

- `php artisan m:controller Admin Robot`

You can pre-answer yes/no steps from CLI so the wizard does not ask again.

Example:

- `php artisan m:controller Admin Robot --create-request=yes --search-request=no --select-items=yes`

For all updated `m:*` commands, unspecified steps are still asked by default.
If you want non-interactive defaults for unspecified steps, add:

- `--skip-questions`

Option values for y/n parameters:

- `yes`, `y`, `1`, `true`
- `no`, `n`, `0`, `false`

The wizard can generate (and/or overwrite) optional classes:

- `<Name>CreateRequest`
- `<Name>SearchRequest`
- `<Name>Action`
- `<Name>Query`
- `<Name>Resource`

Then it always generates `<Name>Controller`.

### Auto-generated tests by layer

When creating classes with `m:controller`, `m:action`, `m:query`, `m:request`, `m:resource`, the generator now creates one small corresponding test per class:

- `Controller` => `Modules/<Module>/Tests/Feature/Http/Controllers/<Name>ControllerTest.php`
- `Action` => `Modules/<Module>/Tests/Unit/Http/Actions/<Name>ActionTest.php`
- `Query` => `Modules/<Module>/Tests/Unit/Http/Queries/<Name>QueryTest.php`
- `Request` => `Modules/<Module>/Tests/Unit/Http/Requests/<Name>RequestTest.php`
- `Resource` => `Modules/<Module>/Tests/Unit/Http/Resources/<Name>ResourceTest.php`

If you need controller tests in Unit for isolated class checks, use:

- `m:controller ... --controller-test-layer=unit`
- `m:module-test <Module> --type=controller --controller-layer=unit`

No aggregate "full-layer" test file is generated by these make commands.

Shared app model tests convention:

- app models in `App\\Models\\...` are tested in root tests (`tests/Unit/Models/*`), not inside `Modules/*/Tests`
- module test generators (`m:module-test`) focus on module classes (controller/action/query/request/resource)
- `m:model` enforces one test file per model class at `tests/Unit/Models/<Model>ModelTest.php`

## Generate tests for existing classes in a module

- `php artisan m:module-test {Module}`

Examples:

- `php artisan m:module-test Admin`
- `php artisan m:module-test Admin --type=controller`
- `php artisan m:module-test Admin --type=action --class=CreateUser`
- `php artisan m:module-test Admin --type=all --force`

Options:

- `--type=all|controller|action|query|request|resource`
- `--class=<ClassName>` generate only one class test (suffix can be omitted)
- `--force` overwrite test file if it exists
- `--include-base` include base module classes such as `AdminAction`, `AdminQuery`
- `--controller-layer=feature|unit` choose controller test target layer (default: `feature`)

### Enable CRUD/search/detail actions during generation

The wizard asks per endpoint:

- `Add create endpoint trait/wrapper?`
- `Add update endpoint trait/wrapper?`
- `Add delete endpoint trait/wrapper?`
- `Add search endpoint trait/wrapper?`
- `Add detail endpoint trait/wrapper?`

For each selected endpoint, it runs `add:action` to wire:

- single-purpose controller trait (`EntryCreateTrait`, `EntryUpdateTrait`, etc.)
- wrapper method in controller
- route entry in `Modules/<Module>/Routes/api.php`

### Enable select-items API during generation

At the end of the wizard, you will be asked:

- `Enable select-items API for this controller?`

If you choose **yes**, the wizard runs `add:select-item` automatically.

## Add select-items API to an existing CRUD

Sometimes CRUD is implemented first, and select-items is added later. Use:

- `php artisan add:select-item Admin Task`

## Add create-or-update API (1-1 relationship settings)

Dùng cho setting 1-1 (vd mail server / hotel). Code sinh ra **rõ ràng**, không qua Service hay trait.

```bash
php artisan m:add-createOrUpdate Admin Hotel serverMail settingMailServer
```

Tự sinh theo rule (không cần nhập thêm):

| Input | Sinh ra |
|-------|---------|
| `Hotel` + `settingMailServer` | `HotelSettingMailServerRequest` |
| `settingMailServer` | route `hotels/{id}/setting-mail-server` |
| parent model `Hotel` | `match-key` = `hotel_id` |
| `serverMail` relationship | related model từ `HasOne` |

Nếu thiếu model hoặc relationship → báo lỗi + hint.

### Điều kiện bắt buộc

1. Parent model (`App\Models\Platform\Hotel`)
2. Related model (`HotelServerMail`) có `belongsTo()`
3. Parent có HasOne — tên **lowerCamelCase**:

```php
public function serverMail(): HasOne
{
    return $this->hasOne(HotelServerMail::class);
}
```

### Sinh ra gì

- `HotelSettingMailServerRequest` = `{Controller}{Method}Request`
- `HotelController::settingMailServer($id, $request)` → `$this->action->settingMailServer(...)`
- `HotelAction::settingMailServer($id, $data)` — `findOrFail()->serverMail()->updateOrCreate()` với `$data` từ Request, **không** chuẩn hóa/encrypt ở Action
- `HotelSettingMailServerRequest` — rules theo fillable related model (table `hotel_server_mails`); coder tùy chỉnh required/nullable
- Route: `PUT hotels/{id}/setting-mail-server` (prefix plural, segment kebab của method)

Option: `--force` ghi đè request đã có.

## Add one controller action to an existing controller

Use this to add one endpoint at a time without regenerating the controller:

- `php artisan add:action Admin Robot create`
- `php artisan add:action Admin Robot search`

Supported actions:

- `create`
- `update`
- `delete`
- `search`
- `detail`

What it does (best-effort, idempotent):

- ensures required dependencies exist (`Action`, `Query`, `Request` as needed)
- patches controller imports + trait usage + wrapper method
- patches module API routes to include the selected endpoint

What it does (best-effort, idempotent):

- Creates:
  - `Modules/<Module>/Http/Requests/<Name>SelectItemRequest.php`
  - `Modules/<Module>/Http/Resources/<Name>SelectItemResource.php`
- Patches:
  - `Modules/<Module>/Http/Queries/<Name>Query.php` to use `SelectItemQueryTrait`
  - `Modules/<Module>/Http/Controllers/<Name>Controller.php` to use `SelectItemControllerTrait` + wrapper method
  - `Modules/<Module>/Routes/api.php` to add a `select-items` endpoint (creates a minimal group if needed)

Options:

- `--force` overwrite the generated request/resource files.

### After running add:select-item

If the generator cannot confidently guess the model class, it will create the request with an empty model and print a warning.

Fix by editing:

- `Modules/<Module>/Http/Requests/<Name>SelectItemRequest.php`

and set:

- `protected string $model = App\Models\<Name>::class;`

## Example: FE request/response

Endpoint convention (example for robots):

- `POST /admin/robots/select-items`

Body example:

```json
{
  "filter": { "name": "abc" },
  "key": "id",
  "name": ["name"],
  "info": ["type", "status", "company.name"]
}
```

Response item shape:

```json
{
  "key": 123,
  "name": "Robot A",
  "info": {
    "type": "x",
    "status": 1,
    "company": { "name": "ACME" }
  }
}
```

Example:

- `php artisan m:auth-api Admin User`
